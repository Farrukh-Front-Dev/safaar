UZBRON BACKEND LOGIKASI - BATAFSIL TUSHUNTIRISH
================================================

Fayl maqsadi
------------

Bu fayl UzBron backend qanday ishlashini oddiy tilda tushuntiradi.
Uni dasturchi bo'lmagan odam ham o'qib, "foydalanuvchi tugmani bosganda
backend ichida nima bo'ladi?" degan savolga javob olishi kerak.

Backend texnik nomi:
  @agoda/backend

Texnologiya:
  NestJS + TypeScript + PostgreSQL/Prisma + vaqtinchalik in-memory demo data.

Backend porti:
  localhost:4000

Asosiy API prefix:
  /v1

Eski frontendlar uchun moslashgan prefix:
  /api

Misol:
  http://localhost:4000/v1/health
  http://localhost:4000/api/health


1. KATTA RASM: BACKEND NIMA QILADI?
-----------------------------------

UzBron loyihasida uchta frontend bor:

1. web-user
   Oddiy mijozlar uchun sayt.
   Mijoz mehmonxona yoki avtobus qidiradi, bron qiladi, to'laydi,
   profilini ko'radi, supportga yozadi.

2. web-partner
   Hamkorlar uchun kabinet.
   Mehmonxona egasi yoki avtobus kompaniyasi o'z xonalari, reyslari,
   bronlari va moliyasini boshqaradi.

3. web-admin
   Super admin panel.
   Admin foydalanuvchilarni, hamkorlarni, bronlarni, to'lovlarni,
   refundlarni, kontentni, supportni va sozlamalarni boshqaradi.

Uchala frontend bitta backendga ulanadi.
Ya'ni user, partner, admin uchun alohida backend yo'q.
Bitta NestJS API hammasiga javob beradi.

Oddiy hayotiy o'xshatish:
  Frontend - mijoz ko'radigan oyna.
  Backend - oynaning ortidagi ofis.
  Database - ofisdagi daftar yoki arxiv.
  Controller - resepshndagi xodim, so'rovni qabul qiladi.
  Service - ichkaridagi mutaxassis, haqiqiy ishni qiladi.
  Module - bitta bo'lim: auth bo'limi, booking bo'limi, admin bo'limi.


2. REQUEST QANDAY O'TADI?
-------------------------

Masalan, web-admin login qiladi.

1. Brauzerdagi frontend so'rov yuboradi:
   POST /api/auth/admin/login

2. main.ts so'rovni ko'radi.
   Agar URL /api bilan boshlangan bo'lsa, uni ichkarida /v1 ga aylantiradi.
   Ya'ni:
     /api/auth/admin/login
   ichkarida:
     /v1/auth/admin/login
   bo'lib ishlaydi.

3. NestJS route topadi.
   Bu so'rov auth.controller.ts ichidagi adminLogin methodiga tushadi.

4. ValidationPipe body ni tekshiradi.
   Body degani frontend yuborgan JSON.
   Masalan:
     {
       "username": "admin",
       "email": "admin@uzbron.uz",
       "password": "admin"
     }

   DTO ichida ruxsat berilmagan field bo'lsa, backend xato qaytaradi.
   Shuning uchun oldin "property username should not exist" chiqqan edi.
   Keyin AdminLoginDto ichiga username qo'shildi.

5. Controller service ni chaqiradi.
   Controller o'zi biznes qaror qilmaydi.
   U faqat:
     "AuthService, mana data, login qilib ber"
   deydi.

6. AuthService tekshiradi.
   Admin uchun login:
     admin
   parol:
     admin
   bo'lishi kerak.

7. Backend birinchi bosqichda 2FA challenge qaytaradi.
   Ya'ni:
     "Login to'g'ri, endi 2FA kod kirit"
   Hozir demo 2FA kodi:
     000000

8. Frontend avtomatik yoki foydalanuvchi orqali ikkinchi so'rov yuboradi:
   POST /api/auth/admin/verify-2fa

   Body:
     {
       "challenge_id": "...",
       "code": "000000"
     }

9. AuthService code 000000 ekanini tekshiradi.

10. To'g'ri bo'lsa token beradi.
    Token - keyingi so'rovlarda "men adminman" deb isbotlash uchun kerak.


3. NESTJS FAYLLARINI QANDAY O'QISH KERAK?
-----------------------------------------

NestJS odatda har bir bo'limni 3 ta asosiy faylga ajratadi.

1. *.module.ts
   Modul fayli.
   Bu fayl NestJSga:
     "mana shu controller va service bir bo'limga tegishli"
   deb aytadi.

2. *.controller.ts
   Endpointlar ro'yxati.
   Masalan:
     @Get('users')
   degani:
     GET /users so'rovini qabul qil.

   @Post('login')
   degani:
     POST /login so'rovini qabul qil.

   Controller vazifasi:
     - so'rovni qabul qilish;
     - URL param, query, body ni olish;
     - service methodini chaqirish;
     - natijani qaytarish.

3. *.service.ts
   Asosiy biznes logika.
   Masalan:
     - parolni tekshirish;
     - bron yaratish;
     - narx hisoblash;
     - statusni o'zgartirish;
     - database yoki demo data bilan ishlash.

4. dto/*.dto.ts
   DTO - frontend yuboradigan data shakli.
   Masalan login uchun:
     email, username, password.

   DTO kerak, chunki backend noto'g'ri fieldlarni ushlab qoladi.

5. common/*
   Barcha modullar ishlatadigan umumiy kod.
   Masalan:
     roles.guard.ts - kimga ruxsat borligini tekshiradi.
     actor.ts - requestdagi user/admin/partner kimligini aniqlaydi.
     api-response.interceptor.ts - javob formatini bir xil qiladi.
     http-error.filter.ts - xatolarni bir xil formatda qaytaradi.


4. KODDAGI BELGILARNI ODDIY TILDA TUSHUNISH
-------------------------------------------

@Controller('auth')
  Bu class auth yo'nalishidagi endpointlarni boshqaradi.
  To'liq URL odatda:
    /v1/auth/...

@Get('health')
  Brauzer/server GET so'rov yuborsa ishlaydi.

@Post('admin/login')
  Login kabi ma'lumot yuboriladigan so'rov.

@Patch('profile')
  Bor ma'lumotning bir qismini yangilaydi.

@Delete(':id')
  Biror obyektni o'chirish yoki o'chirilgan deb belgilash.

:id
  URL ichidagi o'zgaruvchi qism.
  Masalan:
    /bookings/abc123
  bu yerda id = abc123.

@Body()
  Frontend yuborgan JSON body ni oladi.

@Query()
  URL oxiridagi filterlarni oladi.
  Masalan:
    /hotels?city_id=city-tashkent

@Param('id')
  URL ichidagi id ni oladi.

constructor(private readonly service: Service)
  Bu class ichiga boshqa service ni ulash.
  Oddiy tilda:
    "menga shu yordamchini ber, men undan foydalanaman".

return
  Method frontendga javob qaytarayotgan joy.

throw new NotFoundException(...)
  Backend xato qaytaradi.
  Masalan "topilmadi".

async / await
  Database kabi kutish kerak bo'lgan ishlarni tartib bilan bajarish.

{}
  Obyekt.
  Masalan:
    { id: '1', name: 'Ali' }

[]
  Ro'yxat.
  Masalan:
    [hotel1, hotel2, hotel3]

;
  TypeScriptda buyruq tugaganini bildiradi.
  Odam tili bilan "shu gap tugadi" degani.


5. SERVER QANDAY START BO'LADI? main.ts
---------------------------------------

Fayl:
  apps/backend/src/main.ts

Bu backendning kirish eshigi.
Server ishga tushganda birinchi shu fayl ishlaydi.

main.ts qiladigan ishlar:

1. NestFactory.create(AppModule)
   NestJS ilovasini yaratadi.

2. ConfigService orqali .env ni o'qiydi.
   Masalan:
     PORT
     API_PREFIX
     CORS_ORIGINS
     DATABASE_URL
     SWAGGER_ENABLED

3. API prefixni oladi.
   Default:
     v1

4. helmet() yoqiladi.
   Bu HTTP xavfsizlik headerlarini qo'shadi.

5. JSON body limiti qo'yiladi.
   Hozir limit:
     1mb

6. CORS yoqiladi.
   CORS degani:
     qaysi frontendlar backendga murojaat qila oladi.

7. /api prefix moslashuvi bor.
   Eski frontend /api ishlatsa ham backend /v1 ichida qabul qiladi.

8. app.setGlobalPrefix(apiPrefix)
   Barcha endpointlar /v1 bilan boshlanadi.

9. ValidationPipe yoqiladi.
   Muhim sozlamalar:
     whitelist: true
       DTO ichida yo'q fieldlarni olib tashlaydi.

     forbidNonWhitelisted: true
       DTO ichida yo'q field bo'lsa xato beradi.
       Masalan oldingi xato:
         property username should not exist

     transform: true
       Query/body qiymatlarini kerakli tipga aylantirishga yordam beradi.

10. ApiResponseInterceptor yoqiladi.
    /v1 javoblarini:
      { success: true, data: ..., meta: ... }
    ko'rinishiga keltiradi.

11. HttpErrorFilter yoqiladi.
    Xatolarni:
      { success: false, error: ..., meta: ... }
    ko'rinishiga keltiradi.

12. Swagger yoqiladi.
    API hujjati:
      /v1/docs

13. app.listen(PORT)
    Serverni 4000 portda ishga tushiradi.

8. AUTH LOGIKASI
----------------

Papka:
  apps/backend/src/auth

Asosiy fayllar:
  auth.controller.ts
  auth.service.ts
  dto/auth.dto.ts

Auth degani:
  kim kiryapti, kim ekanini isbotlayaptimi, qaysi rolga ega.

8.1. User OTP login

Endpointlar:
  POST /v1/auth/user/send-otp
  POST /v1/auth/user/verify-otp

Oqim:

1. User telefon raqamini yuboradi.
2. Backend telefonni normalize qiladi.
   Masalan:
     901234567
   bo'lsa:
     +998901234567
3. Development muhitida OTP doim:
     111111
4. Production muhitida random 6 xonali kod bo'ladi.
5. Backend OTP kodni ochiq saqlamaydi.
   SHA-256 hash qilib Map ichida saqlaydi.
6. OTP muddati:
     5 daqiqa
7. User code yuboradi.
8. Code hash bilan solishtiriladi.
9. To'g'ri bo'lsa user topiladi yoki yangi user yaratiladi.
10. Access token va refresh token qaytariladi.

8.2. Partner OTP alias

Endpointlar:
  POST /v1/auth/otp/request
  POST /v1/auth/otp/verify

Bu endpointlar partner frontend moslashuvi uchun qo'shilgan.
OTP logic user OTP bilan bir xil, lekin verify bo'lganda PARTNER token beradi.

8.3. Partner email login

Endpoint:
  POST /v1/auth/partner/login

Hozir demo logika:
  parol = password

To'g'ri bo'lsa:
  role = PARTNER
  organization_id = demo-partner-org-id
  partner_role = owner

Muhim:
  Bu hozir to'liq real DB password tekshiruvi emas.
  Keyingi real bosqichda partner_users.password_hash bilan tekshiriladi.

8.4. Admin login

Endpointlar:
  POST /v1/auth/admin/login
  POST /v1/auth/admin/verify-2fa

Hozirgi demo login:
  login = admin
  parol = admin
  email sifatida = admin@uzbron.uz ham qabul qilinadi.

Login to'g'ri bo'lsa backend 2FA challenge qaytaradi.

2FA demo kodi:
  000000

2FA to'g'ri bo'lsa:
  role = SUPER_ADMIN
  actorId = demo-admin-id

Admin frontend login body yuboradi:
  username
  email
  password

Shuning uchun AdminLoginDto ichida username ham, email ham ruxsat qilingan.

2FA body yuboradi:
  challenge_id
  code

Shuning uchun Verify2faDto ichida challenge_id ham ruxsat qilingan.

8.5. Token formati

Hozir token real JWT emas, development mock token:

  mock-access.<actorId>.<role>.<sessionId>
  mock-refresh.<actorId>.<sessionId>

Masalan:
  mock-access.demo-admin-id.SUPER_ADMIN.abc...

CurrentActor shu tokenni o'qib:
  id
  role
  actorType
  organizationId
  sessionId
aniqlaydi.

8.6. Session

AuthService ichida sessionsStore bor.
Token berilganda session shu ro'yxatga yoziladi.

Refresh:
  refresh token sessionsStore ichida topilsa yangi token beradi.

Logout:
  hozir demo javob qaytaradi.

Logout-all:
  hozir demo javob qaytaradi.

Sessions:
  actor uchun sessionlar ro'yxatini qaytaradi.

15.1. Hotel booking yaratish

POST /v1/bookings/hotel

Oqim:

1. Actor aniqlanadi.
   Ya'ni bookingni qaysi user yaratmoqda.

2. hotel_id olinadi.

3. room_id olinadi.

4. Hotel va room borligi tekshiriladi.

5. check_in va check_out sanalaridan nights hisoblanadi.
   Agar sana noto'g'ri bo'lsa, default 1 tun.

6. rooms soni olinadi.
   Default 1.

7. Narx:
   subtotal = room.base_price * nights * rooms

8. createBooking chaqiriladi.

9. createBooking ichida:
   id yaratiladi;
   booking_number yaratiladi;
   status = pending;
   currency = UZS;
   commission_amount = subtotal * 12%;
   partner_payable = subtotal - commission;
   expires_at = 15 daqiqadan keyin;
   item ichiga hotel, room, sana, guestlar yoziladi.

10. Booking ro'yxat boshiga qo'shiladi.

11. Status historyga "created" yoziladi.

12. Payment yaratiladi.

13. Frontendga:
    booking
    payment
    qaytadi.

15.2. Bus booking yaratish

POST /v1/bookings/bus

Oqim:

1. trip_id olinadi.
2. Trip borligi tekshiriladi.
3. Frontend seats yuborgan bo'lsa, ular olinadi.
4. Seats yuborilmasa, birinchi available seat olinadi.
5. Seatlar mavjud va available ekanligi tekshiriladi.
6. Agar seat band bo'lsa:
   SEAT_NOT_AVAILABLE
7. subtotal seat narxlari yig'indisi.
8. Booking yaratiladi.
9. Tanlangan seatlar held qilinadi.
10. held_by_booking_id yoziladi.
11. held_until booking.expires_at bo'ladi.
12. Payment yaratiladi.

15.3. Booking cancel

POST /v1/bookings/:id/cancel

Oqim:
  booking topiladi;
  agar allaqachon cancelled bo'lsa xato;
  status = cancelled;
  cancelled_at yoziladi;
  reason yoziladi;
  audit log yoziladi;
  status historyga yoziladi.

15.4. Cancel preview

34. ENG KO'P UCHRAYDIGAN XATOLAR
--------------------------------

1. "property username should not exist"

Sabab:
  Frontend username yuboryapti, backend DTO ichida username yo'q.

Yechim:
  DTOga username qo'shish yoki frontend yubormasligi kerak.

Bu admin login uchun tuzatildi.

2. "property challenge_id should not exist"

Sabab:
  Frontend verify-2fa body ichida challenge_id yuboryapti,
  backend Verify2faDto ichida challenge_id yo'q.

Bu ham tuzatildi.

3. 403 Forbidden

Sabab:
  Endpoint role talab qiladi, token/headerda mos role yo'q.

Yechim:
  To'g'ri token yuborish.
  Admin uchun SUPER_ADMIN token.
  Partner uchun PARTNER token.

4. Bron topilmadi

Sabab:
  booking id noto'g'ri yoki in-memory server restart bo'lgan.

5. DB data ko'rinmayapti, lekin mock data ko'rinyapti

Sabab:
  DATABASE_URL yo'q yoki Postgres ulanmayapti.
  PostgresService null qaytarganda fallback ishlaydi.

6. DBda data bor, frontda bo'sh

Tekshirish kerak:
  backend DATABASE_URL to'g'rimi;
  seed shu DBga yurganmi;
  endpoint Postgresdan o'qiyaptimi yoki in-memorymi;
  frontend /api yoki /v1 URLga to'g'ri uryaptimi;
  auth token bormi.
