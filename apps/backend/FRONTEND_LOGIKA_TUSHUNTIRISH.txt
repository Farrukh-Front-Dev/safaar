UZBRON FRONTEND LOGIKASI TUSHUNTIRISH
=====================================

Bu fayl 3 ta frontend panel qanday ishlashini oddiy tilda tushuntiradi:

1. web-user    - mijozlar sayti, port 3000.
2. web-partner - hamkor/mehmonxona egasi paneli, port 3001.
3. web-admin   - super admin paneli, port 3002.

Asosiy fikr:

- Uchala frontend alohida Next.js app.
- Backend bitta: NestJS API, odatda port 4000.
- Frontend tugmalari ikki xil ish qiladi:
  1. Faqat sahifa almashtiradi yoki ekrandagi holatni o'zgartiradi.
  2. Backendga HTTP so'rov yuboradi.
- Har bir panelda "page" sahifani chizadi, "component" tugma/input/modalni chizadi,
  "api client" backendga boradigan so'rovni yuboradi, "store" esa vaqtinchalik
  frontend xotirasini saqlaydi.


UMUMIY FRONTEND TUZILISHI
=========================

Monorepoda frontendlar shu joylarda turadi:

- apps/web-user
- apps/web-partner
- apps/web-admin

Har bir app ichida Next.js App Router ishlatiladi. Ya'ni sahifa fayllari `app`
papkasida turadi. Masalan:

- apps/web-user/app/[lang]/(main)/hotels/page.tsx
- apps/web-partner/app/(dashboard)/reservations/page.tsx
- apps/web-admin/app/(dashboard)/users/page.tsx

Oddiy tushuntirish:

- `page.tsx` - brauzerda ko'rinadigan asosiy sahifa.
- `layout.tsx` - sahifaning umumiy qobig'i, masalan sidebar/topbar.
- `components` yoki `_components` - tugma, jadval, modal, forma kabi mayda qismlar.
- `lib/api` yoki `_lib/api` - backend bilan gaplashadigan kod.
- `store` - frontend xotirasi. Masalan login bo'lgan user yoki demo reservationlar.
- `actions.ts` - Next.js server action. Forma submit bo'lganda server tomonda ishlaydi.

BACKEND JAVOBINI FRONTEND QANDAY O'QIYDI
========================================

web-user:

- `apiFetch` degan umumiy funksiya bor.
- Backenddan kelgan javob agar `{ data: ... }` ko'rinishida bo'lsa, ichidagi
  `data` ni olib beradi.
- Xato bo'lsa `ApiRequestError` chiqaradi.
- Token kerak bo'lsa `Authorization: Bearer <token>` qo'shadi.

web-admin:

- `axios` ishlatiladi.
- Request ketishidan oldin cookie ichidagi `admin_token` olinadi.
- Token bo'lsa `Authorization: Bearer <token>` header qo'shiladi.
- Har doim demo header ham qo'shiladi:
  - `x-user-role: SUPER_ADMIN`
  - `x-user-id: demo-admin-id`
- Agar backend 401 qaytarsa:
  - `admin_token` cookie o'chiriladi.
  - Browser `/login` sahifasiga yuboriladi.

web-partner:

- `fetch` ustidan `request<T>` wrapper bor.
- Token bo'lsa `Authorization: Bearer <token>` qo'shadi.
- Lekin partner paneldagi asosiy amallar hozir `data-store` ichida lokal yuradi.

WEB-USER LOGIN OQIMI
====================

Asosiy fayllar:

- apps/web-user/components/auth/LoginForm.tsx
- apps/web-user/lib/auth/actions.ts
- apps/web-user/lib/api/auth.ts
- apps/web-user/lib/auth/session.ts

Login ikki bosqichli:

1. Telefon raqam kiritiladi.
2. OTP kod kiritiladi.

"Kod yuborish" tugmasi bosilganda:

- LoginForm formani `requestOtpAction` ga yuboradi.
- `requestOtpAction` telefon raqam borligini tekshiradi.
- Keyin `sendOtp(phone)` chaqiriladi.
- `sendOtp` backendga boradi:
  - Method: POST
  - URL: `/auth/user/send-otp`
  - Body: `{ phone }`
- Backend OTP yuboradi yoki demo kod qaytaradi.
- Forma ikkinchi bosqichga o'tadi.

"Tasdiqlash" tugmasi bosilganda:

- LoginForm formani `verifyOtpAction` ga yuboradi.
- `verifyOtpAction` telefon va kodni oladi.
- `verifyOtp(phone, code)` chaqiriladi.
- Backendga so'rov ketadi:
  - Method: POST
  - URL: `/auth/user/verify-otp`
  - Body: `{ phone, code }`
- Backend access token, refresh token va user ma'lumotini qaytaradi.
- `setSession` cookie saqlaydi.
- Cookie nomi: `uzbron_session`
- Keyin foydalanuvchi kerakli sahifaga redirect qilinadi.

"Chiqish" tugmasi bosilganda:

- `logoutAction(locale)` ishlaydi.
- `uzbron_session` cookie o'chiriladi.
- User `/${locale}` bosh sahifaga yuboriladi.

Oddiy tilda:

- User telefon kiritadi.
- Backend telefonni tekshiradi va kod yuboradi.
- User kod kiritadi.
- Backend "ha, bu shu user" deydi.
- Frontend cookie saqlaydi.
- Keyingi bron, profil, sevimli kabi so'rovlarda shu cookie ichidagi token ishlatiladi.

Asosiy sahifalar:

- apps/web-admin/app/(dashboard)/cms/banners/page.tsx
- apps/web-admin/app/(dashboard)/cms/news/page.tsx
- apps/web-admin/app/(dashboard)/cms/pages/page.tsx
- apps/web-admin/app/(dashboard)/cms/offers/page.tsx

Banners sahifasi ochilganda:

- `AdminApi.getCmsBanners()` ishlaydi.
- Backendga so'rov:
  - GET `/admin/cms/banners`

News sahifasi ochilganda:

- `AdminApi.getCmsNews()` ishlaydi.
- Backendga so'rov:
  - GET `/admin/cms/news`

Pages sahifasi ochilganda:

- `AdminApi.getCmsPages()` ishlaydi.
- Backendga so'rov:
  - GET `/admin/cms/pages`

Offers sahifasi ochilganda:

- `AdminApi.getCmsOffers()` ishlaydi.
- Backendga so'rov:
  - GET `/admin/cms/offers`

Search/filter tugmalari:

- Ko'p joyda local filter sifatida ishlaydi.

Add/edit/delete ko'rinishidagi tugmalar:

- Agar sahifada bor bo'lsa, hozir real backend mutationga to'liq ulanmagan
  bo'lishi mumkin.
- Asosiy real ulangan qism - ro'yxatni backenddan o'qish.

WEB-ADMIN SUPPORT
=================

Asosiy fayl:

- apps/web-admin/app/(dashboard)/support/page.tsx

Support sahifasi ochilganda:

- `AdminApi.getTickets()` ishlaydi.
- Backendga so'rov:
  - GET `/admin/support/tickets`

Ticket bosilganda:

- `handleOpenTicket(ticket)` ishlaydi.
- Tanlangan ticket local statega yoziladi.
- Ticket messages olinadi:
  - GET `/admin/support/tickets/:id`

Search input:

- Local filter.

Status filter:

- Local filter.

Priority filter:

- Local filter.

Status o'zgartirish:

- `handleStatusChange` ishlaydi.
- Hozir local state ichida ticket status o'zgaradi.
- Backendga PUT/PATCH ketmaydi.

Message yozib "Yuborish" bosilganda:

- `handleSendMessage` ishlaydi.
- Yangi message local statega qo'shiladi.
- Input tozalanadi.
- Backendga POST ketmaydi.

Muhim:

- Ticket ro'yxati va messages backenddan o'qiladi.
- Status update va message send hozir local ishlaydi.

WEB-ADMIN TOPBAR VA SIDEBAR
===========================

Topbar logout tugmasi:

- Zustand auth store tozalanadi.
- `admin_token` cookie o'chiriladi.
- User `/login` sahifasiga yuboriladi.

Notification icon:

- Notification dropdown ochadi.
- `markAllAsRead`, `clearAll`, `markAsRead` kabi local store amallari ishlaydi.
- Backendga so'rov ketmaydi.

Sidebar parent item:

- Ochiladi/yopiladi.
- Backendga so'rov ketmaydi.

Sidebar child link:

- Kerakli sahifaga olib boradi.
- Masalan:
  - `/users`
  - `/partners/list`
  - `/bookings/hotels`
  - `/finance/overview`
  - `/support`


WEB-ADMIN QISQA TUGMA XARITASI
==============================

- Login, eye icon:
  - Parolni ko'rsatadi/yashiradi.

- Login, "Kirish":
  - `/auth/admin/login` ga POST.
  - Kerak bo'lsa `/auth/admin/verify-2fa` ga POST.
  - Cookie saqlaydi.
  - `/dashboard` ga o'tadi.

- Logout:
  - Cookie o'chadi.
  - Auth store tozalanadi.
  - `/login` ga o'tadi.

- Dashboard quick action:
  - Faqat kerakli sahifaga link.

- Users search/status:
  - Local filter.

- Users export:
  - Excel download.

- User row:
  - `/users/:id` ga o'tadi.

- Partners search/status/type:
  - Local filter.

- Partner row:
  - `/partners/:id` ga o'tadi.

- Partner requests "Ko'rish":
  - Modal ochadi.

- Partner requests "Tasdiqlash/Rad etish":
  - Hozir backend update qilmaydi, modal yopiladi.

- Hotel booking row:
  - `/bookings/:id` ga o'tadi.

- Bus booking row:
  - `/bookings/:id` ga o'tadi.

- Booking detail customer link:
  - `/users/:customerId` ga o'tadi.

- Withdrawals "Tasdiqlash":
  - Local status approved.

- Withdrawals "Rad etish":
  - Local status rejected.

- Support ticket row:
  - Ticket detail/messages ochadi.

- Support status change:
  - Local status o'zgaradi.

- Support message send:
  - Local message qo'shiladi.

- Audit search:
  - Local filter.

- Settings save:
  - Local save/alert.

ODDIY XULOSA
============

web-user:

- Mijoz paneli backendga eng ko'p real ulangan.
- Login, hotel, bus, booking, profile, favorite amallari backendga boradi.
- Search tugmalari ko'pincha avval URLni o'zgartiradi, keyin page backenddan
  ma'lumot oladi.

web-partner:

- Hozircha demo/local panelga o'xshaydi.
- Tugmalarning ko'pi databasega yozmaydi.
- Backendga to'liq ulash uchun data-store funksiyalarini API mutationlar bilan
  almashtirish kerak.

web-admin:

- Ro'yxatlarni backenddan o'qishga ulangan.
- Login backend orqali ishlaydi.
- Ba'zi admin actionlar hali local:
  - partner request approve/reject,
  - withdrawal approve/reject,
  - support reply/status,
  - settings save.

Eng muhim farq:

- "Ma'lumot ko'rinishi" ko'p joyda backenddan.
- "Ma'lumotni o'zgartirish" esa ayrim joylarda hali frontend local state bilan.
